> 合并方式：相同或高度相关的问题已归并；每项均补充“是否属实”和“建议合理性”判断。

## 判断标准

- **属实**：当前代码或页面实现能支撑原报告描述，属于真实缺陷、明显体验短板或已实现业务中的缺口。
- **部分属实**：原报告指出的方向存在，但描述有偏差，或需要产品规则进一步定义后才能判断是否必须修改。
- **不属实/过时**：当前代码已经实现，或原报告引用的代码片段不存在，或描述与现有实现不符。
- **建议合理性：高**：建议能修复真实 bug、安全风险、数据口径问题或核心业务闭环。
- **建议合理性：中**：建议能明显改善体验、可维护性或大数据量场景，但不影响当前核心可用性。
- **建议合理性：低**：偏长期产品能力或锦上添花功能，适合进入迭代池。
- **建议合理性：需产品确认**：技术上可做，但要先明确业务口径，否则可能改错方向。

## 六、总改进建议

### 6.1 第一阶段：先修真实 bug 和安全边界

1. 修复学生端作品详情权限，防止非作者访问未审核作品。
2. 修复作品取消点赞导致数据库 `likes` 变负的问题。
3. 修复教师端密码重置驳回原因未传参。
4. 统一学生练习统计口径，确保总题数、已练习、正确率和今日练习来自同一课程范围。
5. 为消息通知任务增加精确练习入口，避免学生从任务跳转后还要自行寻找题目。
6. 梳理学习进度写入路径，让 `StudentProgress.learn_progress` 不再只是被读取。

### 6.2 第二阶段：补齐核心学习和教学闭环

1. 学生端优先实现答题进度保存、错题重练、课程/资料搜索、学习进度展示。
2. 教师端优先补题库分页、导入失败详情、课程/资料分页、作业完成报告分页和列宽统一。
3. 管理员端优先补公共课程同步状态和资料预览，批量导入作为后续增强。
4. 统一图片失败占位、PDF 预览和上传进度方案，避免学生端、教师端、管理员端各自重复实现。

### 6.3 第三阶段：进入产品增强和工程化

1. 学生端的作品推荐、评论、分享、成长档案导出、班级平均对比属于长期产品能力，建议在核心闭环稳定后规划。
2. 教师端的成绩分布图、题目正确率统计、批量审核、动态快捷入口可以按教师真实使用频率排序。
3. 管理端的平台统计、教师操作日志和批量学生管理适合形成独立管理端迭代。
4. 前后端类型可以逐步用 OpenAPI 生成或建立统一接口类型约定，优先解决当前已经发现的 `LikedProject`、`WrongQuestion` 类型漂移。

### 6.4 实施注意事项

1. 不要把所有建议一次性改完。三份原报告混合了 bug、体验增强和长期功能，必须按优先级拆分。
2. 涉及数据库结构、任务答题记录、评论、学习进度、公共课程同步状态的改动，需要先写设计或实施计划。
3. 涉及接口语义变化时，同步检查前端 API 类型、后端统一响应和测试夹具。
4. 对“不属实/过时”的原报告项，不建议进入修复计划，避免重复建设或改坏已有功能。
