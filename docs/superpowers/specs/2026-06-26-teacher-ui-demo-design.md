# 教师端后台 UI 改造 Demo 设计说明

> 日期：2026-06-26  
> 阶段：仅 demo 原型，不改业务代码  
> Demo 文件：`output/teacher-ui-demo/index.html`

## 要解决的问题

教师端当前页面功能基本完整，但后台管理体验仍偏分散：概览页只展示局部信息，列表页筛选、表格、空状态、弹窗和操作反馈的密度不够统一。此次先制作独立 demo，用于确认教师端 Vue 页面改造和后台管理 UI 优化方向。

本阶段目标不是上线新功能，而是让用户先看到可点击、可浏览的完整原型，再决定是否进入正式 Vue 页面改造。

## 涉及页面和模块

Demo 选取 3 类代表界面：

1. 教师工作台概览：对应 `frontend/src/views/teacher/TeacherDashboard.vue`。
2. 学生成绩表格：对应 `frontend/src/views/teacher/TeacherStudents.vue`。
3. 课程管理列表和弹窗：对应 `frontend/src/views/teacher/TeacherCourses.vue`。

Demo 同时保留教师端 10 个固定导航入口，入口命名遵守现有教师端规范：概述、课程管理、班级管理、发布作业、学生成绩、作业完成、作品审核、资料管理、学生管理、题库管理。

## 范围内

- 制作独立静态 HTML 原型，放在 `output/teacher-ui-demo/index.html`。
- 使用中文页面文案、按钮、提示、空状态和表格列名。
- 使用后台管理风格：稳定、清晰、易扫描，避免营销式 hero 和装饰性布局。
- 覆盖导航切换、概览指标、作业列表、待审作品、筛选栏、成绩表格、右侧详情、空状态、课程列表、公共课程、新建课程弹窗和 Toast 反馈。
- 保持视觉方向与项目现有靛青主色、宣纸白背景、宋体标题气质兼容。

## 范围外

- 不修改 `frontend/src/views/teacher/` 下任何正式 Vue 页面。
- 不调整前端路由、API 封装、权限逻辑、导出逻辑。
- 不修改后端接口、数据库、初始化脚本或测试夹具。
- 不引入新 UI 框架或新的状态管理方式。
- 不把 demo 当作最终实现；正式改造需要另写实施计划并经确认后执行。

## 设计方向

推荐方向是“密度适中的工作台化后台”：

- 左侧导航仍为教师端主入口，强调当前页状态和待审数量。
- 顶部保留搜索、返回学生端、教师身份信息，减少重复页面标题操作。
- 概览页优先显示待处理事项，而不是只放统计卡。
- 表格页使用筛选栏 + 表格 + 详情侧栏，减少频繁弹窗跳转。
- 课程页使用主列表 + 右侧公共课程区域，明确“我的课程”和“可添加课程”的关系。
- 危险操作在正式实现时继续使用确认弹窗；失败状态必须给出可理解中文提示。

## 验收方式

Demo 阶段验收：

- 浏览器打开 `output/teacher-ui-demo/index.html`，首屏不是空白。
- 左侧“概述 / 学生成绩 / 课程管理”可切换。
- 其他导航入口点击后有中文提示，不产生错误。
- 学生成绩页可以点击表格行更新右侧学生详情。
- 学生成绩页可以切换空状态并返回列表。
- 课程管理页可以打开、关闭、提交“新建课程”弹窗。
- 页面在桌面和窄屏下无明显文字重叠、按钮溢出、乱码。

正式实现阶段验收需另行补充：

- 前端至少执行 `npm run build`。
- 教师端页面需浏览器打开验证中文文案、表格、弹窗、Toast、空状态。
- 不引入无关业务改动。

服务器不需要拉代码、不需要重启服务、不需要重新构建前端、不需要执行数据库迁移、不需要调整环境变量或 Nginx 配置。
