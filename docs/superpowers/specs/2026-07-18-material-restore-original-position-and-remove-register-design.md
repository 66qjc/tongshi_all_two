# 资料原位恢复与注册下线设计

> 状态：设计已确认，待写实施计划与落地  
> 日期：2026-07-18  
> 来源：服务器前端操作验收 G-05、G-10  
> 关联：`docs/superpowers/specs/2026-07-15-unified-soft-delete-business-rules-design.md`（资料恢复语义补充）

## 一、目标

1. **G-05 资料原位恢复**：管理员从回收站恢复资料时，资料必须回到删除时所在课程位置（原阶段）。若因阶段级联删除导致原阶段已不存在，则在同课程（或重挂目标课程）下自动重建同名阶段并挂回，而不是落到“未分类/未分阶段”。
2. **G-10 注册下线**：平台不再提供学生自助注册。账号仅由管理员创建/导入教师、教师 Excel 导入学生产生。注册页面与后端注册接口一并移除。

## 二、范围

### 2.1 范围内

| 主题 | 内容 |
|---|---|
| G-05 删除路径 | 单份资料软删；阶段级联软删资料时写入阶段定位快照，不再为删阶段而抹掉可恢复定位信息 |
| G-05 恢复路径 | 原阶段存在则挂回；不存在则按快照在同课程/目标课程重建同名阶段后挂回；原课程已清理仍走现有 `target_course_id` |
| G-05 数据 | 资料表增加删除时阶段快照字段（可空）；库结构兼容 |
| G-10 前端 | 删除 `RegisterView`、路由 `/register`、前端 register API 与 store 方法，清理残留链接 |
| G-10 后端 | 删除 `POST /register`、`register_user` 及仅服务注册的请求模型 |
| 测试 | 原位恢复用例；注册改为“接口不存在”；相关回归不破坏 |
| 文档/验收 | 同步恢复语义、注册下线说明与验收清单表述 |

### 2.2 范围外

- 不做完整「课程阶段进回收站」产品
- 不改变班级、作业、题目、用户、作品的恢复语义
- 不改登录、改密、密保、忘记密码
- 不改管理员建教师、教师导入学生的主流程
- 不为旧客户端保留注册兼容占位接口

## 三、现状与根因

### 3.1 资料恢复不回原位（G-05）

当前行为：

1. 业务读路径过滤 `deleted_at is not null` 的资料，删除后不可见（正确）。
2. `restore_resource` 对资料主要清除 `deleted_at` / `deleted_by`；仅在 `course_id` 为空时要求 `target_course_id` 重挂课程。
3. `delete_stage(..., cascade_materials=True)` 会：
   - 软删阶段下活跃资料；
   - 将 `material.stage_id = None`；
   - **物理删除**阶段行。
4. `Material.stage_id` 外键为 `ON DELETE SET NULL`，阶段删除后定位丢失。

因此“恢复=重新可见”，不等于“回到删除时的目录位置”。

### 3.2 账号来源与注册（G-10）

- 学生：教师端 Excel 导入（`import_students_from_excel`，可挂班级）。
- 教师：管理员单建 + Excel 批量导入。
- 自助注册：前端 `/register` + 后端 `POST /api/register`，且仅允许 `student`。
- 登录页等常规导航几乎无注册入口，但路由与接口仍存在，与“仅导入建号”的产品策略冲突。

## 四、G-05 详细设计

### 4.1 数据：删除时阶段快照

在 `materials` 表增加可空快照字段（实现命名以落地为准，语义固定）：

| 字段 | 含义 |
|---|---|
| `deleted_stage_id` | 软删瞬间的原阶段 ID |
| `deleted_stage_name` | 软删瞬间的阶段名称 |
| `deleted_stage_sort_order` | 软删瞬间的阶段排序 |

约束：

- 仅在软删发生时写入；活跃资料不依赖这些字段。
- 快照不受阶段行物理删除与 `ON DELETE SET NULL` 影响，是恢复原位的权威信息。
- 恢复成功并完成阶段挂回后清空快照，避免干扰下次删除。
- 历史已在回收站、删除时无快照的资料：恢复成功，但不保证原位；审计/结果可注明“无阶段快照”。

### 4.2 删除路径

#### 单份资料软删

在置 `deleted_at` 前：

1. 若当前 `stage_id` 有值且阶段存在：写入三份快照字段。
2. 若本就未分阶段：快照保持空。
3. 不主动清空 `stage_id`（即便后续阶段删除导致 DB 置空，快照已保留）。

#### 阶段级联删除（教师/管理员，`cascade_materials=true`）

1. 对每个活跃资料：先写阶段快照，再软删。
2. **禁止**再执行“仅为脱钩而 `material.stage_id = None` 且不写快照”的逻辑。
3. 阶段行仍可按现有产品物理删除（本轮不做阶段回收站）。
4. 业务读路径继续只看 `deleted_at`，已软删资料不可见。

### 4.3 恢复路径（`restore_resource` 对 materials）

顺序固定：

1. 校验资料仍在回收站（现有规则）。
2. **课程挂载**
   - `course_id` 非空且课程活跃：使用原课程。
   - `course_id` 为空：必须提供活跃 `target_course_id`（现有）。
3. **阶段挂回（新增）**
   - 若当前 `stage_id` 仍指向**目标课程下存在**的阶段：确认/保持挂回。
   - 否则若存在 `deleted_stage_name`（或完整可用快照）：
     - 在目标课程查找同名阶段；
     - 找到则复用，不新建；
     - 找不到则创建阶段（name=快照名，sort_order=快照排序或课程内合理默认），再挂回。
   - 若无任何阶段快照：`stage_id` 保持空（未分类），恢复不失败。
4. 清除 `deleted_at` / `deleted_by`，并清空快照字段。
5. 审计 details 记录：是否重建阶段、目标阶段 ID/名称、是否重挂课程、是否因无快照未能原位。

### 4.4 边界矩阵

| 场景 | 行为 |
|---|---|
| 原阶段仍在 | 直接挂回，不新建阶段 |
| 原阶段已物理删除 | 同课程重建同名阶段后挂回 |
| 目标课程已存在同名阶段 | 复用，不建重复 |
| 删除时本就未分阶段 | 恢复后仍未分阶段 |
| 原课程已清理，恢复时指定新课程 | 在新课程下按快照阶段名建/挂阶段 |
| 旧数据无快照 | 恢复成功，阶段可空；兼容不报错 |

### 4.5 前端

- 回收站恢复交互保持：仅在资料 `course_id` 为空时要求输入目标课程 ID。
- 可选增强成功提示（非阻塞）：“已恢复并回到阶段「xxx」”或“已恢复并重建阶段「xxx」”。

### 4.6 与既有软删规格的关系

补充而非推翻 `2026-07-15-unified-soft-delete-business-rules-design.md`：

- “资料恢复仅恢复资料自身”细化为：**恢复资料自身 + 恢复其删除时的阶段挂载位置**；不恢复其它无关资源。
- 不把 `CourseStage` 纳入七类回收站资源。
- 课程同批恢复子资源规则不变；本设计聚焦**单份资料**及**阶段级联导致的资料软删**。

## 五、G-10 详细设计

### 5.1 产品策略

| 角色 | 建号方式 |
|---|---|
| 教师 | 管理员单建 / Excel 批量导入 |
| 学生 | 教师 Excel 导入（可挂班级） |
| 任意角色 | 不提供自助注册 |

### 5.2 删除清单

**前端**

- 删除 `frontend/src/views/RegisterView.vue`
- 删除路由 `/register`
- 删除 `frontend/src/api/auth.ts` 中 `RegisterPayload` 与 `register()`
- 删除 `frontend/src/stores/auth.ts` 中 `register` 方法与导出
- 全仓扫尾删除指向注册的文案/链接

访问 `/register` → 现有兜底 `/:pathMatch(.*)*` → 404 页。不单独做“注册已关闭”页。

**后端**

- 删除 `auth_routes.py` 的 `POST /register`
- 删除 `auth_service.register_user`
- 删除无其它引用的 `RegisterRequest`
- 清理仅服务注册的 import

**明确保留**

- 登录、退出、改密、密保、忘记密码
- 管理员创建/导入教师
- 教师导入学生
- 用户软删与回收站恢复

### 5.3 兼容与安全

- 旧书签 `/register`：前端 404。
- 旧客户端 `POST /api/register`：无路由，不可建号。
- 不提供 410/占位注册接口。

### 5.4 验收与文档表述

- 验收清单 G-10：改为“不适用/已下线”，或改为“确认无注册入口且直访 404；账号仅导入创建”。
- 对外/内部说明中若仍写“可注册”，改为“账号由管理员或教师导入创建”。

## 六、测试计划

### 6.1 后端

1. 单删资料 → 恢复 → `stage_id` 与删除前一致，且出现在原阶段下。
2. 阶段 `cascade_materials=true` → 资料软删且快照有名称 → 恢复 → 同课程存在同名阶段且资料挂上。
3. 先 cascade 删阶段，再手动新建同名阶段 → 恢复复用，不重复建。
4. 无快照旧数据 → 恢复不报错，阶段可空。
5. 原课程清理后带 `target_course_id` 恢复 → 在新课程下按快照名挂/建阶段。
6. 阶段 cascade 回归：资料读路径不可见；删除阶段本身仍成功。
7. `POST /api/register` 不可用（404 或等价无路由）。
8. 登录、改密、教师导入、学生导入既有测试保持通过。

### 6.2 前端

1. `npm run type-check`、`npm run build` 无注册残留引用。
2. 手工：`/register` 404；登录页无注册引导。
3. 手工（G-05）：删资料/阶段级联后回收站恢复，课程详情中位置正确。

## 七、风险与部署影响

### 7.1 风险

| 风险 | 处理 |
|---|---|
| 历史回收站资料无快照 | 兼容恢复，不保证原位；文档写明 |
| 同名阶段语义冲突 | 固定“目标课程内按名称复用”，不自动拆分 |
| 库表加列 | 走项目现有 schema 兼容/初始化；字段可空 |
| 误删登录相关代码 | 删除清单仅限注册面；登录改密导入单测兜底 |

### 7.2 服务器部署影响（落地后必须再写进修改记录）

| 变更 | 服务器操作 |
|---|---|
| 后端恢复逻辑 + 资料表快照列 | 拉代码；按现有方式做库结构兼容/迁移；重启后端 |
| 前端删除注册 | 重新构建前端并发布静态资源 |

## 八、验收口径

1. 课程详情删除资料 → 列表不可见 → 回收站恢复 → 仍在原阶段下。
2. 阶段级联删除资料后，回收站恢复资料 → 同课程出现同名阶段，资料在其下。
3. 原课程已物理清理、恢复时指定新课程 → 在新课程下按快照阶段名挂回（可重建阶段）。
4. 访问 `/register` 为 404；`POST /api/register` 不可用；登录与导入建号可用。
5. 相关自动化测试与前端构建通过；修改记录含服务器部署影响说明。

1. G-05 数据字段与删除写快照。
2. G-05 恢复挂回/重建阶段 + 测试。

4. G-10 后端注册删除 + 测试改写。
5. G-10 前端删除与构建验证。
6. 文档、验收清单、修改记录与部署说明同步。
