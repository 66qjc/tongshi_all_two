# 公开学习馆与书站式学习页设计

## 一、背景

当前平台原本以学生、教师和管理员三类角色为主，学习资料主要服务已加入课程的学生。新的阶段目标是把平台先调整为更公众化的 AI 通识学习平台：游客也能看到大部分公开学习内容，进入页面后愿意继续浏览和学习，同时保留现有学生端、教师端、管理员端能力，后续仍可继续用于正式教学。

用户明确的近期优先级是：

1. 首页补充“深度”两个字和学校名字。
2. 游客能看到大部分公开学习内容。
3. 展示页面更美观，有“眼前一亮”的吸引力。
4. “学”页面参考 `book.guyuehome.com` 的书站式学习体验。
5. 所有上传资料都应能服务学习展示，但旧的学生、教师和后台功能暂时保留。

## 二、优先级分层

当前任务拆为 P0 和 P1 两层，实施时不得让 P1 挤占 P0。

### P0：游客访问与首页公开化

P0 是本轮最高优先级，目标是让服务器上的公开页面先能稳定展示出来：

1. 首页第一屏补充学校名字和“深度”表达。
2. `/learn` 对游客开放，不再强制跳登录。
3. 游客能看到公开课程、已发布课时和公开资料信息。
4. 前端公开页面样式在服务器部署后能正常展示，不能只在本地好看。
5. 修复 P0 相关页面中文乱码、空状态和错误提示。
6. 保持教师端、管理员端和已登录学生功能不被破坏。

### P1：书架与图文资料馆

P1 是 P0 稳定后的体验升级，目标是把“学”页面从普通课程列表提升为更有吸引力的公开学习馆：

1. `/learn` 做成“AI 通识书架”或“图文资料馆”。
2. 一本“书”第一阶段可以复用公开课程，不急着新增图书数据表。
3. 点击一本书后进入类似古月居书站的阅读页：左目录、中正文、右资料。
4. PDF 不再只是单独展开，而是作为图文导读、章节内容和配套资料的一部分展示。
5. 后台仍通过教师端或管理员端上传资料，不把内容维护改成纯静态资源。

## 三、参考对象

参考站点为古月居图文教程站：`https://book.guyuehome.com/`。

可借鉴点：

1. 内容组织采用“教程书站”结构，左侧目录清晰，正文阅读优先。
2. 首页和引言页聚合课程视频、图文教程、源码、问答和社区链接，便于学习者从一个入口进入完整学习路径。
3. 底部和公开资料显示该站由 Material for MkDocs 生成，说明其本质是静态文档站，不适合直接复制技术栈到当前 Vue + FastAPI 项目。
4. GitHub 参考仓库为 `guyuehome/ros2_21_tutorials`，更适合借鉴内容结构和资源聚合方式，而不是直接复用前后端代码。

因此，本项目不切换到 MkDocs，不复制静态站构建链路，而是在现有 Vue 页面和 FastAPI 接口上实现类似的学习体验。

## 四、当前代码状态

### 已经完成的能力

1. 课程模型已有 `is_public` 字段，可作为公开学习内容源的第一版基础。
2. 已有公共课程管理、教师副本同步、公共课程资料和题库共享相关服务。
3. “学”页 `/learn` 已能展示课程列表、搜索课程、读取课程进度。
4. 课程详情 `/learn/course/:courseId` 已有接近书站的阅读壳：
   - 左侧课程目录。
   - 中间课时正文。
   - 上方课程信息。
   - 课时和资料两个标签。
   - 上一课、下一课导航。
5. 课时阅读器 `LessonReader.vue` 已能解析正文内嵌资料块，并渲染视频、PDF 资料卡片。
6. 资料卡 `MaterialRichCard.vue` 已有封面、摘要、状态和预览入口。
7. 资料预览已有站内弹窗和大文件预览试点，PDF 与视频优先走 `/api/materials/{material_id}/file`。
8. 后端已有课程、课时、资料、阶段、进度等完整基础接口。

### 还没有完成或不适配的内容

1. `/learn` 和 `/learn/course/:courseId` 当前不是公开路由，游客会被跳转到登录页。
2. 后端课程、课时、资料、进度接口都依赖登录用户，游客没有读取公开学习内容的通道。
3. 学生访问课程详情要求加入课程班级，这对“公众化学习平台”不适用。
4. 当前 `/learn` 更像学生课程列表，不像公开学习馆，也缺少专题、学习路径、精选资料等入口。
5. 当前课程详情虽然有目录和正文，但视觉仍偏内部学习系统，不够像公开教程书站。
6. 首页还没有把“深度”和学校名字作为第一屏明确表达。
7. `LearnView.vue`、`CourseDetailView.vue`、`router/index.ts`、`api/http.ts`、部分文档里存在中文乱码表现，影响视觉和可信度，必须作为第一阶段修复项。
8. 上传资料进入学习展示的规则还不清晰，教师上传、管理员公共课程上传和游客展示之间缺少明确边界。

## 五、方案取舍

### 方案 A：只改前端视觉

只把首页和“学”页做漂亮，后端权限不动。

优点：最快，风险最低。

缺点：游客仍看不到真实课程、课时和资料，无法满足“给所有人学习”的核心目标。

结论：不推荐作为本阶段主方案。

### 方案 B：公开学习 API + 书站式学习页

新增只读的公开学习接口，以现有公共课程和已发布课时为内容源。前端把 `/learn` 改为公开学习馆，课程详情改为书站式阅读页。游客可看公开课程、已发布课时、公开资料预览；登录用户额外保留学习进度、练习、作品、消息等能力。

优点：能满足当前目标，复用现有课程、课时、资料和预览能力，权限边界清晰。

缺点：需要同时调整前端、后端和权限边界，必须写实施计划并测试。

结论：推荐。

### 方案 C：重构内容模型为完整知识库

重新设计课程、专题、文章、资料、标签、搜索和推荐体系。

优点：长期扩展性最好。

缺点：范围过大，会拖慢当前最优先目标。

结论：不作为第一阶段目标。

## 六、推荐设计

第一阶段采用方案 B，目标是做成“公开学习馆 + 书站式课程阅读页”。

### 信息架构

1. 首页 `/`
   - 第一屏明确出现学校名字。
   - 平台主张加入“深度”表达，例如“深度 AI 通识学习平台”。
   - 给出进入公开学习馆的主入口。
   - 保留登录、注册、教师和管理员入口。

2. 学习馆 `/learn`
   - 改为游客可访问。
   - 页面定位从“我的课程列表”改为“AI 通识公开学习馆”。
   - 首屏展示精选公开课程、学习路径、资料数量和课程数量。
   - 中段展示公开课程列表，课程卡片强调课程主题、简介、课时数、资料数、学习状态。
   - 下方展示热门资料或最新资料，帮助没有明确课程目标的访客先点开一个资料。
   - 登录学生仍可看到“继续学习”状态，但游客不显示私人进度。
   - P1 再升级为“书架”表达：公开课程以前台“书籍”的形态呈现，保留课程作为底层数据源。

3. 课程阅读页 `/learn/course/:courseId`
   - 改为游客可访问公开课程。
   - 桌面端采用三栏书站式布局：
     - 左侧：课程目录，显示已发布课时。
     - 中间：正文阅读区，优先阅读课时内容。
     - 右侧：本课资料、课程信息、学习建议或相关资源。
   - 移动端改为顶部课程信息 + 抽屉目录 + 正文。
   - 资料标签保留，但从“附属资料列表”升级为“资料库”体验。
   - 游客可浏览正文和公开资料元信息，可打开允许公开预览的 PDF 或视频。
   - 登录学生可继续保存进度；游客不调用进度接口。
   - P1 进一步支持“点击一本书进入阅读”的体验，PDF 作为书内配套资料或章节资料展示，不再以单独文件列表为唯一形态。

4. 教师端和管理员端
   - 第一阶段不重做后台。
   - 教师端现有上传、资料管理、课程管理保持可用。
   - 管理员公共课程继续作为公开内容源。
   - 上传资料只要挂到公开课程或公开课程副本规则允许展示，就进入公开学习展示。
   - 公开学习馆不是纯静态资源站，内容仍由后台上传和管理；静态化只体现在前台阅读体验接近书站。

### 内容来源规则

第一阶段不新增复杂内容模型，使用现有 `Course.is_public` 作为公开判断：

1. 公开课程：`Course.is_public = true`。
2. 公开课时：公开课程下 `Lesson.status = "published"` 的课时。
3. 公开资料：公开课程下的资料，按阶段或未分类展示。
4. 草稿课时不对游客展示。
5. 私有教师课程不对游客展示。
6. 学生私人进度、作业、练习记录、作品提交不对游客展示。

如果后续需要更细粒度控制，可再增加 `Material.is_public` 或 `Lesson.visibility`，但第一阶段先不改数据库结构。

### 接口设计

新增公开学习只读接口，避免直接放开现有教学接口：

1. `GET /api/public/learning/courses`
   - 返回公开课程列表。
   - 支持关键词搜索。
   - 返回课程简介、课时数、资料数、封面信息或首个资料封面。

2. `GET /api/public/learning/courses/{course_id}`
   - 返回公开课程详情。
   - 只允许读取 `is_public = true` 的课程。
   - 返回课程基础信息、阶段、资料和已发布课时目录。

3. `GET /api/public/learning/courses/{course_id}/lessons`
   - 返回公开课程下已发布课时列表。

4. `GET /api/public/learning/materials`
   - 返回公开资料列表，可用于学习馆首页的热门资料或最新资料模块。

5. `GET /api/public/learning/materials/{material_id}/file`
   - 第一阶段可选。
   - 如果实现，必须只允许公开课程下资料文件访问，并继续使用 Nginx 内部跳转。
   - 如果暂不实现，游客只看资料卡片和摘要，登录后再预览文件。

现有登录接口保持不变：

1. `/api/courses*` 继续服务学生、教师和管理员。
2. `/api/courses/{course_id}/progress` 只服务登录学生。
3. `/api/materials/{material_id}/file` 继续服务登录后的授权预览。

### 前端数据流

1. 新增 `frontend/src/api/publicLearning.ts`，封装公开课程、公开课程详情、公开课时和公开资料接口。
2. `/learn` 根据登录状态决定是否额外读取个人进度：
   - 游客：只请求公开学习接口。
   - 学生：请求公开学习接口，必要时再请求个人进度接口。
   - 教师：仍按现有路由守卫进入教师工作台，不展示学生端业务页。
3. `/learn/course/:courseId` 根据公开接口加载课程详情、目录、课时和资料。
4. 保存学习进度时先判断登录状态，游客不保存，不弹错误。
5. 资料预览根据资料是否有可公开预览文件决定按钮状态。

### 视觉方向

设计目标不是做后台感，而是做“可信、清爽、愿意阅读”的公开学习站。

1. 学习馆首屏
   - 不做营销空话和大面积装饰卡片。
   - 用课程知识地图作为第一视觉：AI 通识从“理解 AI、使用 AI、评价 AI、创造 AI、负责任 AI”展开。
   - 主操作是“开始学习”和“浏览资料”。
   - 学校名字和平台定位清晰出现。

2. 课程卡片
   - 每张课程卡展示主题、简介、课时数、资料数和阅读入口。
   - 避免同质化图标网格，课程之间要有内容差异。
   - 使用资料封面、阶段色块或课程主题标签增强识别度。

3. 阅读页
   - 中间正文宽度控制在适合阅读的范围。
   - 左侧目录固定但不遮挡正文。
   - 右侧资料栏突出 PDF、视频和相关资源。
   - 使用清晰标题、段落、引用、资料嵌入块和上一课下一课导航。

4. 中文文案
   - 第一阶段必须修复乱码。
   - 所有按钮、空状态、错误提示都使用正常中文。
   - 游客态文案要自然，例如“登录后可保存学习进度”，不能把游客当成错误状态。

## 七、权限边界

游客只能读取公开学习内容：

1. 可访问 `/learn`。
2. 可访问公开课程详情。
3. 可读取公开课程下已发布课时。
4. 可查看公开资料卡片、摘要、封面和基础信息。
5. 是否可直接预览文件由第一阶段实施计划进一步确认。

游客不能访问：

1. 私有课程。
2. 草稿课时。
3. 学生个人进度。
4. 练习、作业、错题、作品提交、消息中心。
5. 教师端和管理员端。

登录学生保留：

1. 个人学习进度。
2. 已加入课程访问。
3. 练习和任务能力。
4. 作品和成长档案能力。

教师和管理员保留现有后台能力，不因为公开学习馆而放宽管理权限。

## 八、阶段范围

### P0 范围

P0 做：

1. 首页文案和视觉入口优化，必须出现学校名字和“深度”表达。
2. `/learn` 改为公开学习入口，游客可访问。
3. 新增或调整公开学习只读接口，使游客能读取公开课程、已发布课时和公开资料信息。
4. 游客态不请求个人进度接口。
5. 登录学生保留进度保存。
6. 修复 P0 相关页面中文乱码。
7. 确保前端样式可在服务器部署后展示：构建产物、静态资源路径、公开路由刷新、图片资源引用都要纳入验收。
8. 更新 `docs/superpowers/` 修改记录，并明确服务器部署影响。

P0 不做：

1. 不把 `/learn` 完整重做成书架视觉。
2. 不重做教师端或管理员端上传后台。
3. 不改数据库结构。
4. 不开放私有课程给游客。
5. 不实现复杂 PDF 自动解析、自动转章节或自动摘要。

### P1 范围

P1 做：

1. `/learn` 升级为“AI 通识书架”或“图文资料馆”。
2. 公开课程以前台“书籍”形态展示，卡片具备封面、简介、章节数、资料数和阅读入口。
3. `/learn/course/:courseId` 做成更完整的书站式三栏阅读页。
4. PDF、视频和附件以“配套资料”方式嵌入章节和右侧资料栏。
5. 没有图文正文的 PDF 资料，也要通过封面、摘要、页数、资料卡和站内预览形成更美观的展示。
6. 管理员端公共课程继续作为公开内容源；教师端资料上传继续保留，公开发布规则后续单独细化。
7. 增加必要的前端视觉回归或静态检查，避免书架样式在服务器构建后丢失。

P1 不做：

1. 不重构全部内容模型。
2. 不新增推荐算法。
3. 不开放私有课程给游客。
4. 不重做教师端完整上传后台。
5. 不改数据库结构，除非实施时发现公开预览文件必须增加字段。
6. 不把项目改成 MkDocs 或静态站。

## 十、验收标准

1. 未登录游客打开 `/learn` 不跳登录页。
2. 未登录游客能看到公开课程、已发布课时目录和公开资料信息。
3. 未登录游客打开私有课程时返回清晰的不可访问状态。
4. 学生登录后仍能保存课程学习进度。
5. 教师账号仍不能进入学生端业务页面，继续跳转教师工作台。
6. 公开学习页和课程阅读页无中文乱码。
7. 首页出现学校名字和“深度”表达。
8. 课程阅读页桌面端不出现目录遮挡正文，移动端目录可正常打开和关闭。
9. 后端公开接口只返回公开课程、已发布课时和公开课程资料。
10. 前端构建通过，后端受影响测试通过或说明无法验证的原因。
11. P0 上线前必须确认服务器访问公开页面时 CSS、图片、字体、路由刷新和资料封面能正常加载。
12. P1 上线前再验收书架视觉、图文资料馆、书籍卡片和书站式阅读体验。

## 十一、风险与处理

1. 风险：直接放开现有接口可能泄露私有课程。
   - 处理：新增公开学习只读接口，不改变现有教学接口语义。

2. 风险：游客预览文件可能绕过现有资料权限。
   - 处理：公开文件预览接口单独校验资料所属课程是否公开，不复用学生加入课程判断。

3. 风险：课程详情同时服务游客和学生，进度逻辑可能报错。
   - 处理：前端按登录态调用进度接口，游客态不保存进度。

4. 风险：页面视觉只改样式但内容仍空。
   - 处理：以公开课程、课时、资料为真实数据源，页面围绕内容组织。

5. 风险：中文乱码继续扩大。
   - 处理：第一阶段把涉及页面和路由文案统一修正为 UTF-8 正常中文，并用静态检查保护重点文件。

6. 风险：本地前端样式正常，但服务器部署后静态资源路径、路由刷新或构建产物异常。
   - 处理：P0 实施计划必须包含 `npm run build`、静态资源路径检查、生产路由刷新检查和服务器部署影响说明。

## 十二、服务器部署影响

本设计本身只新增文档，不需要服务器修改。

后续按本设计实施代码后，预计服务器需要：

1. 拉取最新前端和后端代码。
2. 重新构建并部署前端静态资源。
3. 重启后端服务以加载新增公开学习接口。
4. 不需要数据库迁移，前提是第一阶段不新增字段。
5. 不需要调整环境变量。
6. 如果实现公开资料文件预览，并继续复用 Nginx 内部跳转，需要确认服务器现有 Nginx `X-Accel-Redirect` 配置仍覆盖资料文件目录。
