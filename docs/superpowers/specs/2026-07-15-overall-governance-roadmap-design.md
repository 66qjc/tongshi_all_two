## 一、目标与边界

本设计把一致性评估中的待优化项整理为面向非技术使用者的四阶段治理路线图。目标不是一次性重构系统，而是让用户先获得可靠的删除与恢复能力，并逐步让发布、并发和长期维护具备可验证的规则。

本设计负责阶段顺序、依赖关系、验收门、风险登记和跨任务接口。它不重复设计七类资源的软删除细节，不罗列服务器命令，也不修改代码、数据库、服务器配置或生产服务。

### 范围内

- 统一现有计划的先后顺序与上线前置条件。
- 明确每个阶段面对用户的结果、完成标准和停止条件。
- 规定阶段之间可交接的稳定事实，避免两个任务分别修改同一条规则。
- 规定维护停服发布的最低保护要求。

### 不在范围内

- 重写 `2026-07-10-remediation-phase-d-soft-delete-cache-db.md` 中七类资源、缓存、索引和启动锁的具体实现。
- 重写 `2026-07-14-single-server-ops-hardening.md` 中备份、磁盘、服务和部署命令。
- 重写 `多人占线计划暂存.md` 中多 worker、连接池、Nginx 和前端高延迟适配方案。
- 在本设计阶段执行服务器变更、发布、数据迁移或代码改动。

## 二、路线选择

项目当前最重要的用户可见问题不是新功能不足，而是同样叫“删除”的操作会产生不同结果：有的资源可恢复，有的直接消失；有的删除后仍在列表、统计或文件入口出现。首阶段因此以数据规则统一为核心。

生产安全仍是上线前提，但不与首阶段争夺开发优先级：删除规则可以先在本地和隔离环境完成实现与验证；实际维护发布前，必须通过本设计第二阶段定义的最低保护门。当前无人使用，允许维护停服，不要求双实例切换或零中断发布。

## 三、四阶段路线图

### 第一阶段：业务数据规则

**面向用户的结果：** 删除资源后不会继续被普通用户看到；误删可以从回收站恢复；恢复不会误带回以前独立删除的数据；历史答题、阅读、图片和关联记录不会因一次删除被提前破坏。

**工作边界：**

- 以阶段 D 的 Task 1 为唯一实现依据，统一用户、课程、班级、作业、作品、资料、题目的正式删除入口。
- 对普通列表、详情、统计、通知目标和文件授权统一使用活跃数据规则；回收站、恢复和最终清理是读取已删除资源的专用入口。
- 明确关联记录保留、同批次级联恢复、题目引用保护和审计日志的规则。
- 以阶段 D 的 Task 2 为依据，统一题干规范化和 SHA-256 候选哈希，完成旧数据升级；哈希只用于辅助发现疑似重复题，不改变现有最终重复判定。
- 本阶段不启用新的 Redis 缓存策略、不调整多 worker、不进行无关的数据模型重构。

**进入条件：** 已冻结本阶段涉及资源的产品语义；现有回收站和软删除字段状态已核实。

**当前事实（2026-07-15）：** 工作区已具备正式入口软删、同批次恢复、purge、活跃读过滤和 SHA-256 哈希主体实现；总体路线图 Task 0 已分类工作区并完成基线。**不得再按“从零实现”推进**；若继续第一阶段，只允许差距验收、补测与文档收口。未通过验收门前不得进入第二阶段发布。

**验收门：**

1. 七类资源经各自正常删除入口后都能在回收站找到，普通页面、文件入口和统计均不可见。
2. 恢复后仅恢复同一删除批次的应恢复数据，独立删除的数据仍保持删除状态。
3. 最终清理前不物理删除主记录或必要关联历史；最终清理是唯一物理删除入口。
4. 每条删除、恢复和最终清理路径均留下可查询审计记录。
5. 创建、编辑和导入题目使用同一题干哈希规则，旧 MD5 或缺失哈希均完成安全升级。
6. 受影响的后端测试、前端构建和回归路径通过。

**交接产物：** 删除生命周期说明、资源活跃过滤约定、文件授权过滤约定、删除相关回归测试清单和审计动作名称。

### 第二阶段：生产安全与可恢复发布

**面向用户的结果：** 维护期间即使发布失败，也能恢复到已知可用版本和已验证的数据状态；线上运行版本不再依赖人工猜测。

**工作边界：**

- 以单机加固计划为唯一实施依据，完成数据库与上传文件备份、恢复演练、磁盘与日志保护、部署健康检查和可回退发布流程。
- 以一致性评估的 P0 为准，收敛唯一发布事实源，明确服务器源码、前端静态产物、Nginx 配置样例和实际配置的对应关系。
- HTTPS、后端环回绑定和端口边界属于本阶段安全子项；实施前需要单独确认域名、证书和网络权限。
- 本阶段允许为发布安排维护停服；停服前必须先通过备份、回退和健康检查门，不以“服务器当前正常”替代验证。

**进入条件：** 第一阶段已在本地或隔离环境通过业务验收，且待发布变更可对应到明确提交。

**验收门：**

1. 数据库和上传文件备份均可在隔离位置恢复，并完成关键记录、文件数量和抽样文件一致性核对。
2. 服务器磁盘、journal 和构建空间达到计划阈值；空间不足时部署被拒绝。
3. 发布脚本从干净、可追溯的源码构建，前端静态文件与构建版本一致，失败时有明确回退入口。
4. 维护发布后首页、健康接口、关键 API 和受控文件访问均通过；失败时不能宣称发布成功。
5. 服务器变更记录写明拉取代码、服务重启、前端构建、数据库迁移、环境变量和配置的实际影响。

**交接产物：** 可恢复发布基线、版本标识、备份与恢复记录、维护发布检查单、线上资源阈值和已确认的网络边界。

### 第三阶段：并发性能与跨进程一致性

**面向用户的结果：** 用户增多时，课程、班级和认证结果仍一致，不会因为命中不同缓存或被不同进程处理而看到过期、越权或不一致的数据。

**工作边界：**

- 以阶段 D 的 Task 3 至 Task 5 为前置实现依据，先修正缓存键、权限检查、DTO 序列化、失效策略、Redis 故障降级、忘记密码共享限流、索引和数据库启动锁。
- 以多人占线计划为扩容依据；只在缓存和启动锁验证完成后评估多 worker、连接池和压测参数。
- 前端超时、重试和加载反馈在真实压测结果表明需要时实施，不把超时调大当作性能已解决。
- 本阶段不重新定义第一阶段的删除和恢复规则；缓存失效只能消费该阶段交接的删除生命周期。

**进入条件：** 第二阶段的可恢复发布能力可用；第一阶段已发布且删除语义稳定。

**验收门：**

1. 缓存键只由稳定业务参数组成，权限检查发生在命中缓存之前，缓存值为可序列化的最终数据。
2. 新增、编辑、删除和恢复后，相关旧缓存被精确失效；Redis 故障时核心业务可降级运行且不会产生日志风暴。
3. 两个进程共享忘记密码失败计数；未启用 Redis 时不得按多 worker 安全语义验收。
4. 数据库兼容升级在同一命名锁内完成，目标索引与 ORM 声明一致，并通过真实 MySQL 测试库验证。
5. 完成压测后，才按实际 CPU、内存、MySQL 连接和 P95 响应时间决定 worker 与连接池规模。

**交接产物：** 缓存命名空间与失效事件表、多 worker 准入结论、压测基线、数据库结构升级验证记录。

### 第四阶段：工程结构与文档

**面向用户的结果：** 后续修改更容易判断影响范围，关键接口字段变化和规则偏差会在上线前被发现，而不是依赖熟悉历史的个人记忆。

**工作边界：**

- 优先把高风险写接口的“业务写入、审计、缓存失效和事务提交”收敛到服务层；不为形式一次性迁移所有只读路由。
- 为登录、分页、详情、文件签名和作业等关键接口补充响应契约检查，逐步降低前后端手工类型漂移。
- 建立正式术语表：`Announcement(type=quiz)` 对应当前产品中的作业；共享题库中的课程是挂载上下文，创建人是编辑归属，不能把课程外键误读为传统所有权。
- 整理长期文档为当前架构事实、领域决策、运维手册和变更记录四类；历史计划保留但不充当当前事实源。

**进入条件：** 前三阶段的稳定事实、验收记录和接口边界已形成。

**验收门：**

1. 高风险写路径只有一个事务拥有者，异常时业务数据、审计和缓存失效不会部分成功。
2. 核心接口字段的破坏性变化能在自动检查中失败。
3. 新增功能不会再引入删除、作业或共享题库的第四套语义。
4. 项目地图、运维说明和变更记录能相互指向同一稳定事实，已完成计划不会被写成进行中。

**交接产物：** 服务层事务约定、接口契约清单、术语表、文档索引和变更记录模板。

## 四、阶段依赖与接口

| 提供阶段 | 稳定接口或事实 | 使用阶段 | 边界规则 |
|---|---|---|---|
| 第一阶段 | 删除、恢复、最终清理、活跃过滤、审计动作 | 第二、三、四阶段 | 后续阶段不得自行新增物理删除入口或绕过活跃过滤。 |
| 第二阶段 | 备份、回退、版本标识、维护检查单 | 第三、四阶段 | 并发和结构改动只能通过可回退发布流程上线。 |
| 第三阶段 | 缓存事件、启动锁、多 worker 准入、压测基线 | 第四阶段 | 文档和契约必须记录缓存与多进程的已验证边界，不能把计划能力写成现状。 |
| 第四阶段 | 术语、事务、契约和文档规范 | 后续所有任务 | 新功能设计必须引用既有术语和生命周期规则。 |

为避免任务覆盖，阶段 D 继续拥有软删除、缓存、哈希、索引和启动锁的实现细节；单机加固计划继续拥有服务器操作步骤；多人占线计划继续拥有扩容参数和前端高延迟适配。总体路线图只定义这些任务的准入顺序与验收结论。

## 五、风险登记

| 风险 | 触发信号 | 控制方式 | 所属阶段 |
|---|---|---|---|
| 软删除改造遗漏旧读写入口 | 删除资源仍能从列表、统计或文件入口访问 | 以正式入口编写回归测试，统一活跃过滤，回收站之外拒读已删除资源 | 第一阶段 |
| 恢复误带回以前删除的数据 | 恢复一个课程后出现不应恢复的子资源 | 仅恢复删除时间和操作者匹配的同批次级联资源 | 第一阶段 |
| 维护发布造成不可恢复的数据或文件损失 | 无可验证备份、恢复失败或磁盘空间不足 | 发布前先备份与恢复演练，空间不足拒绝维护 | 第二阶段 |
| 发布版本事实分裂 | 服务器源码、静态文件和配置无法对应同一版本 | 干净源码构建、版本标识、发布后哈希和健康检查 | 第二阶段 |
| 缓存显示已删或越权数据 | 不同用户或不同请求看到不同结果 | 先完成权限前置、稳定键、DTO 和删除/恢复失效，再启用缓存 | 第三阶段 |
| 多 worker 导致限流或初始化分裂 | 密码重置计数可绕过、并发启动出现 DDL 冲突 | Redis 共享计数与 MySQL 启动锁验证前禁止多 worker 验收 | 第三阶段 |
| 文档再次漂移 | 计划、项目地图和实际代码描述矛盾 | 将当前事实与历史计划分开，并在阶段验收后同步记录 | 第四阶段 |

## 六、总体验收与服务器部署影响

路线图完成不以“所有文档都写完”作为标准，而以以下事实为准：用户数据删除与恢复可解释且可验证；维护发布有可恢复边界；多进程前的缓存和数据库条件已闭合；后续开发能从稳定文档和测试找到正确规则。

本设计本身不需要服务器修改，不需要拉取代码、重启服务、重新构建前端、执行数据库迁移或调整环境变量。后续每个实施阶段必须在自己的修改记录中明确实际服务器部署影响；未经单独确认，不对服务器执行任何配置或服务变更。
