# 学生作品删除与游客作品展示设计

> **日期：** 2026-07-18  
> **状态：** 已确认设计，待写实施计划  
> **范围：** 学生自助删除（待审/驳回）、游客浏览已通过作品、游客匿名点赞  
> **不在本轮：** 上传即全量展示、游客取消点赞、指纹/严格防刷、管理员独立作品台、学生删除已通过作品

## 一、背景与问题

### 现状

| 能力 | 现状 |
|---|---|
| 学生删除作品 | 无学生删除接口与入口；仅可查看「我的作品」，驳回后可修改再提交 |
| 教师删除作品 | 教师「作品审核」可软删除 |
| 作品广场 | 需登录，且只展示 `status = approved` |
| 作品详情 | 需登录；已通过对登录用户可见，未通过仅作者可见 |
| 点赞 | 绑定登录用户 `ProjectLike`，可点赞/取消 |

### 要解决的问题

1. 学生上传后无法自助删除误传/不想保留的待审或驳回作品。  
2. 游客无法浏览作品广场与详情，公开展示能力不足。  
3. 需求讨论中曾出现「只要上传都可展示」，与现网「先审后展」冲突；产品已确认**本轮仍只展示已通过作品**。

## 二、已确认产品规则

| 主题 | 规则 |
|---|---|
| 删除主体 | 学生为主，教师兜底 |
| 学生可删 | 仅自己的 `pending`、`rejected` |
| 学生不可删 | `approved`（已上墙由教师处理） |
| 教师可删 | 保持现有审核侧删除（含已通过） |
| 游客展示范围 | 仅 `approved` 且未软删 |
| 游客能力 | 列表 + 详情 + 匿名点赞 |
| 匿名点赞 | 只增加 `likes`，不记身份，一般不可取消；前端 localStorage 防连点 |
| 审核 | 维持先审后展，不改为上传即展示 |

## 三、方案选择

采用**最小改造（方案 1）**：

- 复用现有软删除与审核状态机。  
- 列表/详情改为可选登录，未登录只读已通过。  
- 游客点赞单独轻量接口，不改造 `ProjectLike` 语义。  

未选方案 2（下架/IP 限流）与方案 3（访客 like 表/可取消）以避免本轮范围膨胀。

## 四、功能设计

### 4.1 学生删除

**入口**

- 「我的作品」列表：`pending` / `rejected` 显示「删除」。  
- 作品详情（作者本人且状态为待审/驳回）：可显示「删除」。  
- `approved`：不显示删除按钮。

**交互**

1. 点击删除 → 中文确认弹窗。  
2. 文案示例：确认删除作品「{标题}」？删除后不可在「我的作品」中恢复，如需再次展示请重新上传。  
3. 确认后调用删除接口；成功 Toast「已删除」，列表移除该项。  
4. 取消则无变更。

**服务端规则**

- 仅登录学生。  
- 作品存在且 `deleted_at is null`。  
- `author_id == 当前用户`。  
- `status in (pending, rejected)`。  
- 通过现有 `soft_delete` 软删除，写审计（操作者为学生本人）。  
- 已通过、非作者、已删除：返回业务错误，前端展示可读中文。

**教师删除**

- 保持 `DELETE /api/teacher/projects/{id}` 与审核页入口不变。  
- 学生删除与教师删除共用软删除语义，读路径均不可见。

### 4.2 游客作品展示

**可见性**

| 状态 | 游客列表 | 游客详情 | 登录非作者 | 作者本人 |
|---|---|---|---|---|
| approved | 可见 | 可见 | 可见 | 可见 |
| pending | 不可见 | 不可见 | 不可见 | 可见 |
| rejected | 不可见 | 不可见 | 不可见 | 可见 |
| 已软删 | 不可见 | 不可见 | 不可见 | 不可见 |

**页面**

- `/create` 作品广场：未登录可打开，数据仍为已通过作品。  
- 作品详情：未登录可打开已通过作品。  
- 提交作品、「我的作品」、重新提交：仍需登录。  
- 详情可保留「登录后提交作品」引导，不阻断浏览。

**接口语义**

- `GET /api/projects`：可选登录；始终只返回已通过且未软删。  
- `GET /api/projects/{id}`：可选登录；  
  - 未登录：仅 `approved` 可访问；  
  - 已登录：保持现有 `get_accessible_project`（已通过或作者本人）。  
- 未登录时 `is_liked` 固定为 `false`（或等价缺省），点赞数仍返回。

### 4.3 游客匿名点赞

**行为**

1. 游客点击点赞 → `POST` 游客点赞接口。  
2. 仅 `approved` 且未软删可赞。  
3. 服务端对 `Project.likes` 原子 `+1`，不写 `ProjectLike`。  
4. 不支持取消。  
5. 前端以 `localStorage` 键（如 `guest_liked_{projectId}`）标记已赞：  
   - 已标记则按钮展示已赞且不再请求；  
   - 清除缓存或换设备可能再次点赞（可接受）。  

**登录用户**

- 继续使用现有 `POST /api/projects/{id}/like`（可赞可取消，写 `ProjectLike`）。  
- 游客接口与登录接口分离，避免匿名写入 `user_id` 外键。

**明确不保证**

- 严格一人一赞、防刷、跨设备去重。

## 五、接口草案

| 方法 | 路径 | 鉴权 | 说明 |
|---|---|---|---|
| `DELETE` | `/api/projects/{project_id}` | 登录学生 | 作者删除自己的待审/驳回作品 |
| `GET` | `/api/projects` | 可选登录 | 已通过作品分页列表 |
| `GET` | `/api/projects/{project_id}` | 可选登录 | 详情；游客仅已通过 |
| `POST` | `/api/projects/{project_id}/like` | 登录用户 | 现有可切换点赞 |
| `POST` | `/api/projects/{project_id}/guest-like` | 无需登录 | 匿名 `likes+1`，不可取消 |
| `DELETE` | `/api/teacher/projects/{project_id}` | 教师 | 现有教师删除，不变 |

**错误示例（中文 message）**

- 学生删已通过：`已通过作品不可删除，请联系教师处理`  
- 学生删他人作品：`只能删除自己的作品`  
- 游客赞非公开作品：`作品不存在` 或 `作品不可点赞`  
- 未登录访问待审详情：`作品不存在`（不泄露审核中内容）

## 六、前端改动范围

- `frontend/src/api/project.ts`：学生删除、游客点赞 API；列表/详情在无 token 时可请求。  
- `frontend/src/views` 中「我的作品」相关页：删除按钮与确认。  
- `CreateView.vue` / `ProjectDetailView.vue`：未登录可浏览；游客点赞 UI 与 localStorage。  
- 路由：确认 `/create`、作品详情对游客不必强制整页跳登录（若现有全局守卫拦截，需对只读路径放行）。  
- 教师审核页删除：不改产品语义。

## 七、后端改动范围

- `project_routes.py`：新增学生删除；列表/详情改为可选登录；新增 `guest-like`。  
- `project_service.py`：  
  - `delete_own_project`（或扩展删除并校验作者与状态）；  
  - 游客可读详情校验；  
  - `guest_like_project` 原子加一。  
- 鉴权：复用项目已有「可选登录」模式（若已有 helper 则复用，不新造体系）。  
- 软删除与审计：与教师删除一致走 `soft_delete`。  
- 测试：学生删除权限矩阵、游客列表/详情过滤、游客点赞计数、登录点赞回归。

## 八、数据与兼容

- **不改表结构**（本轮）。  
- `Project.status` 仍为 `pending | approved | rejected`（及现有默认）。  
- `Project.likes` 继续为计数；游客赞与登录赞都增加该计数。  
- 历史已通过作品对游客立即可见（接口放开后）。  
- 无需数据库迁移。

## 九、验收标准

1. 学生可删除自己的待审作品，删除后「我的作品」与公开读路径均不可见。  
2. 学生可删除自己的驳回作品。  
3. 学生删除已通过作品被拒绝，UI 无删除入口。  
4. 教师仍可删除权限范围内作品（含已通过）。  
5. 游客打开作品广场，仅见已通过作品。  
6. 游客打开已通过详情成功；直接访问待审/驳回详情失败且不泄露内容。  
7. 游客点赞后计数 +1；同浏览器再次进入显示已赞且不重复请求连点。  
8. 登录用户点赞/取消点赞行为与改前一致。  
9. 文案无乱码；危险删除有确认弹窗。

## 十、服务器部署影响（实施后预期）

- 需要拉取代码。  
- 需要重启后端。  
- 需要重新构建并部署前端。  
- 不需要数据库迁移。  
- 不需要新增生产环境变量。  
- 不需要 Nginx 结构性调整（除非现有把 `/api/projects` 误配成仅认证，实施时按实际配置核对）。

## 十一、风险与后续

| 风险 | 处理 |
|---|---|
| 游客点赞可刷 | 本轮接受；后续可加 IP 时间窗或下架能力（方案 2） |
| 清缓存后重复赞 | 产品已接受 |
| 「上传即可展示」再提 | 需另开需求，改列表过滤与审核语义 |
| 学生误以为能删已通过 | UI 不展示删除，接口再次拦截 |

## 十二、决策记录

- 删除权：学生 + 教师兜底（非仅管理、非三角色齐开）。  
- 学生删除范围：待审 + 驳回；已通过不可删。  
- 游客展示：仅已通过。  
- 游客能力：列表 + 详情 + 匿名点赞。  
- 匿名点赞：轻量只加不减，不记身份。  
- 总体方案：最小改造方案 1。
